Deploying to Railway

A step-by-step for putting one shared instance online — plus a straight answer on whether Vercel can host this as-is (it can't, and here's why).

Railway — yes

Good fit, zero code changes

Runs a persistent container. Add two files (a Dockerfile and a .dockerignore), connect the repo, done. ~30–45 min.

Vercel — not as-is

Architecture mismatch

Vercel runs short-lived serverless functions with a read-only disk. This app is a long-running server that keeps state in memory and writes files. See section 5.

1. Why Railway is the right shape

The whole app is one long-running process (uvicorn backend.main:app) that, at startup, runs the ingestion pipeline + CRM scoring, loads everything into memory, and then serves both the UI and the API. It also writes to disk when you log a CRM note.

Railway (like Render or Fly.io) runs exactly that: a container that stays up, holds memory, and has a writable filesystem. It builds straight from your GitHub repo and hands you a public URL.

The nice part: the port is set by the start command, not baked into the Python, and the app already serves the UI + API from one process. So Railway needs no changes to your code — only two new files that describe how to build and run the container.

2. What you need to add (2 files)

Both go in the repo root. Nothing else in the codebase changes.

Dockerfile required

Two stages: build the React UI with Node, then run the Python server — solves the "needs both Node and Python" wrinkle cleanly.

# ---- Stage 1: build the React frontend ----
FROM node:20-slim AS frontend
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build            # produces dist/

# ---- Stage 2: python runtime ----
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
COPY --from=frontend /app/dist ./dist   # bring in the built UI
# Railway injects $PORT; bind 0.0.0.0 so it's reachable
CMD ["sh", "-c", "uvicorn backend.main:app --host 0.0.0.0 --port ${PORT:-8000}"]

.dockerignore required

Keeps the build fast and clean (don't ship your local venv or node_modules; the UI is rebuilt inside the image).

node_modules
.venv
.venv-1
dist
.git
__pycache__
*.pyc
.DS_Store
Why dist/ is in .dockerignore: it's gitignored and rebuilt inside Stage 1, so you never ship a stale local copy. The one exception is the shortcut path (section 6) where you skip Docker entirely.

3. Deploy it

  1. Commit the two files and push the repo to GitHub. (dist/ can stay gitignored — Docker builds it.)
  2. Go to railway.app → New Project → Deploy from GitHub repo → pick this repo. Railway detects the Dockerfile automatically — no build command to configure.
  3. Add environment variables (Variables tab) — see section 4. You can skip this and it still runs (in fallback mode).
  4. Generate a public URL: Settings → Networking → Generate Domain. Railway gives you a *.up.railway.app address.
  5. Open the URL. First build takes a couple minutes; after that, every git push auto-redeploys.

4. Environment variables

VariableEffect
PORTautoSet by Railway; the start command already reads it. Don't set it yourself.
ANTHROPIC_API_KEYoptionalSet it → real AI (scoring + briefs), billed to your Anthropic account. Leave it out → free deterministic fallback; the app still fully works.
LLM_MODELoptionalOverride the model (default claude-opus-4-8).
LLM_ENABLEDoptionalSet to 0 to force the fallback even when a key is present.

5. Why Vercel won't work as-is

Vercel is built for two things: serving a static frontend, and running short, stateless serverless functions. This app is neither of those — it's a stateful, always-on server. Three concrete blockers:

What the app doesWhy Vercel can't (as-is)
Runs a startup pipeline once and keeps the results in memory (app.state) Serverless functions are created and destroyed per request. There's no persistent process to hold that state, and the startup work would re-run on every cold start (including the CRM AI calls).
Writes files when you log a CRM note (output/, data/crm_notes/) Vercel's function filesystem is read-only except a temporary /tmp that vanishes after the request. Note-saving would fail or never persist.
Serves the UI and the API from one long-lived server Vercel expects the frontend as static files and the backend as discrete functions — not a single uvicorn process you keep running.
It's an architecture mismatch, not a language one. Vercel can run Python — the blocker is the always-on, stateful, file-writing shape of this app, which is the norm for a FastAPI service and the opposite of serverless.

If you specifically want Vercel in the mix

The only clean way is to split the app: host the React frontend on Vercel (static — that part it does beautifully), and run the FastAPI backend on Railway/Render. Then point the frontend's /api calls at the backend's URL and re-enable CORS. That's a small refactor, not an "as-is" deploy — and for one shared demo instance it's more moving parts than just putting the whole thing on Railway.

6. Two shortcuts & one caveat

Skip Docker (fastest, ~15 min)

Build dist/ locally, commit it (un-gitignore it), and deploy as a native Python service on Railway/Render with start command uvicorn backend.main:app --host 0.0.0.0 --port $PORT. No Dockerfile. Downside: you rebuild + commit dist/ on every UI change.

Persisting CRM notes optional

By default the container's disk is ephemeral, so notes logged in the demo are wiped on each redeploy. If you need them to stick, add a Railway Volume and point the notes path at it. For a demo, ephemeral is usually fine.

Cost caveat with a key set: with ANTHROPIC_API_KEY configured, the 7 CRM scores are computed on every boot and each "Generate Brief" is a live call. Fine for a demo; see DEPLOYMENT.md for the offline-caching fix if it matters.
